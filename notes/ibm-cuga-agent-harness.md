# IBM CUGA — 에이전트 '하네스(harness)'와 cuga-apps 예제 24종

## 1. 핵심 요약
IBM Research가 만든 오픈소스 **에이전트 하네스 CUGA(Configurable Generalist Agent)** 소개 글(2026-06-23). 기존 에이전트 앱 개발은 "일주일치 배관 작업"(모델 연결·도구 어댑터·상태관리 UI)이 필요했는데, CUGA는 **계획·실행 루프·도구 호출·상태 배관을 전부 대신 처리**해서 개발자는 *도구 목록 + 프롬프트*만 쓰면 됨. 핵심 주장: **"프레임워크가 아니라 하네스"** → 에이전트 정의는 그대로 두고 배포 환경만 교체. cuga-apps는 작동하는 예제 24종 모음. `pip install cuga`.

## 2. Key Takeaways
- **하네스 vs 프레임워크:** 프레임워크는 *코드 짜는 구조를 강제*, 하네스는 *에이전트 정의는 그대로 두고 실행/배포만 갈아끼움*. 에이전트 앱을 **한 파일**로 유지 가능.
- **성능(작은 모델 + 좋은 하네스):**
  - **AppWorld** 벤치마크 1위 (2025.07~2026.02)
  - **WebArena** 1위 (2025.02~09)
  - 오픈웨이트 **gpt-oss-120b**로 운영 → *"모델 크기보다 하네스 설계"*가 성능을 좌우한다는 메시지.
- **거버넌스 기본 내장(런타임):** Intent Guard, Tool Approval, Tool Guide, Playbook, Output Formatter — 사후 추가가 아니라 처음부터 런타임에 포함 → 프로덕션 갈 때 재작성 불필요.
- **도구 호환:** OpenAPI · MCP · LangChain 함수. 호스팅 MCP 서버 7개(도구 36종: 웹검색·위키·지오코딩·금융데이터 등).
- **통일된 상태 규약:** 모든 도구가 `{"ok": true, "data": ...}` / `{"ok": false, "code": ..., "error": ...}` 형식 → 에이전트가 실패를 선언적으로 처리.
- **추론 모드 전환:** Fast / Balanced / Accurate를 코드 변경 없이 스위치.
- **멀티에이전트:** CugaSupervisor가 전문가 에이전트에 위임.

## 3. 💡 사업 아이디어 (한화 적용 관점)
- **하네스 = 진입장벽 붕괴:** 사내 에이전트 앱을 "한 파일 + 도구목록"으로 만들 수 있으면, 한화 각 계열사(보험·증권·자산운용)가 IT 대형 프로젝트 없이 업무 에이전트를 빠르게 시제품화 가능 → **사업개발 속도** 관점에서 큰 레버리지.
- **거버넌스 내장 = 규제업 적합:** Intent Guard·Tool Approval 같은 정책이 런타임 기본값이라, 금융·보험처럼 *승인/감사/통제*가 필수인 영역에 바로 얹기 좋음 (cf. [[uipath-maestro-case-agentic]]의 human-in-the-loop 논리와 연결).
- **오픈웨이트 + Sovereign Core(주권 실행):** 외부 클라우드/모델에 데이터 안 넘기고 *자체 실행* 가능 → 금융 데이터 보안·국내 규제 대응에 유리. 비용(작은 모델로 1위)도 **재무 관점**에서 매력.
- **재무/세무 자동화 PoC 후보:** 36종 도구 중 '금융 데이터' 포함 → 공시·재무데이터 수집·분석 에이전트, 세무 리서치 에이전트 등 네 관심사 직결 영역에 바로 실험해볼 만함.

## 4. 추가 리서치 / 사례
- **[외부]** 'harness'는 SW의 *test harness*(코드를 실행·검증하게 감싸는 골격)에서 온 말. AI에선 LLM(엔진)을 실제로 일하게 만드는 *실행 계층*(섀시·배선·핸들)을 뜻함. 최근 'agent harness / model harness'로 통용.
- **[외부]** 경쟁/비교 축: LangChain·LangGraph(프레임워크), AutoGen·CrewAI(멀티에이전트), OpenAI Agents SDK. CUGA의 차별점은 "프레임워크가 아닌 하네스 + 거버넌스 런타임 내장 + 벤치 SOTA".

## 5. Q&A / 메모
**Q:** 하네스(harness)가 뭐야?
**A:** LLM을 실제로 '일하게' 감싸는 **주변 인프라/골격(실행 계층)**. LLM 자체는 토큰 예측기라 혼자서는 도구 호출·다단계 계획·실패 복구를 못 함. 그걸 둘러싸 굴러가게 만드는 게 하네스.
- 담당: 계획(planning) + 실행 루프 + 도구 호출 배선 + 상태 관리 + 에러처리/자기수정(reflection) + 출력포맷/정책.
- 비유: **LLM = 엔진, 하네스 = 섀시·배선·핸들·계기판** (엔진만으론 못 달림).
- 어원: SW의 test harness, 원뜻은 말 '마구(馬具)' — 강한 동물을 *제어해 일 시키는 장치*.
- "프레임워크 아니고 하네스" = 프레임워크는 코드 구조를 강제 vs 하네스는 에이전트 정의 그대로 두고 실행/배포만 교체. → 좋은 하네스면 *작은 모델로도* 벤치 1위(글의 핵심 주장).
